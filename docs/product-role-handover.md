# AI Desk Pet 产品角色接管说明（Codex Product Director）

> 文档状态：当前有效（产品管理执行文档）
>
> 接管日期：2026-03-13
>
> 目标：在 Claude 离岗后，保持产品方向、需求优先级和版本节奏连续。

## 1. 角色定义

本角色在仓库内承担“产品负责人/产品总监”职责，负责：

- 维护产品目标与版本路线，避免需求漂移
- 将业务目标拆解为可交付任务（含验收标准）
- 在体验、性能、成本之间做权衡决策
- 管理需求优先级（P0/P1/P2）和迭代节奏
- 推动发布门禁：功能完整性、回归质量、打包可用性

本角色不替代工程实现，但对以下输出负责：

- 每个版本要解决的核心问题
- 每个任务的“完成定义”（DoD）
- 是否进入发布候选（Candidate Build）

## 2. 文档基线与裁决顺序

产品判断与需求裁决必须按以下顺序读取文档：

1. `technical-documentation.md`（当前实现真值）
2. `README.md`（当前对外范围与交付方式）
3. `docs/product-roadmap-2026.md`（中长期规划与北极星指标）
4. `requirements.md`（体验优化需求与验收项）
5. `docs/qa-report-2026-03-11.md`（质量与风险现状）

冲突处理规则：

- 涉及“当前是否已实现”的冲突，以 `technical-documentation.md` 为准
- 涉及“未来做什么”的冲突，以 `docs/product-roadmap-2026.md` 为准
- 涉及“是否可交付”的冲突，以 QA 报告和实际校验结果为准

## 3. 当前产品快照（接管时）

当前结论：

- 产品形态：桌面端常驻 AI 桌宠（macOS/Windows/Linux）
- 版本阶段：v1.x 可验证产品基线，重点是“体验收口 + 性能收口 + 发布收口”
- 交互基线：单击宠物可直接展开环形动作菜单，聊天输入默认隐藏按需展开
- AI 能力：内置零配置聊天、每日限流、短期与长期记忆、跨会话历史恢复
- 养成能力：状态衰减、成长/进化、成长日记、主动行为、专注模式统计
- 发布状态：本地 DMG 可构建；正式 macOS 外部分发仍缺签名与 notarization

已完成的关键批次（基于 v2 指令与 QA 记录）：

- 状态可视化、记忆感知升级、回归问候、主动行为扩充
- 宠物命名、跨会话聊天持久化、长期记忆提取与注入
- 圆形菜单稳定性与动效优化
- 成长日记、心情可视化、输入框体验优化、错误人格化提示

## 4. 当前待解决问题（产品视角）

接管后优先收口项：

- 巡游轨迹自然度仍需提升（移动行为“像活物”但还不够稳定）
- 启动和常驻资源占用需继续下降（长期运行体验）
- 不同角色运动手感差异化不足（角色辨识度受限）
- 正式发布链路未闭环（macOS 签名与 notarization）
- 缺少更系统的桌面端手工回归机制（发布风险偏高）

## 5. 接管后两阶段计划

### 阶段 A（2 周）：v1.2.1 收口版

目标：将“可用”推进到“可稳定试用”。

范围：

- 优化巡游路径与转向过渡，降低随机感和突兀感
- 建立启动/常驻资源采样基线（冷启动、5 分钟、30 分钟）
- 建立最小手工回归清单（交互、聊天、语音、显隐、成长、打包）
- 完成 macOS 正式签名与 notarization 的材料和流水线校验

发布门禁：

- `npm run check` 通过
- 桌面主链路烟测通过
- 打包成功且安装可启动
- 回归清单无 P0/P1 阻塞

### 阶段 B（3-4 周）：v1.3 互动扩展版

目标：提升“新鲜感”和“日活驱动”。

范围（沿用 roadmap）：

- 轻量小游戏（本地题库）
- 心愿清单与番茄钟联动提醒
- 天气互动升级与节日感知
- 设置面板结构化重构（按常用度分层）

发布门禁：

- 不引入高成本常驻开销
- 不破坏现有交互基线和低打扰原则

## 6. 产品决策原则（执行时强约束）

- 情感优先于功能堆叠
- 存在感优先于复杂交互
- 主动行为遵守“不打扰”原则
- 默认零配置可用，配置项仅做增强
- 每个新增能力必须附带可验收标准

## 7. 需求流转模板（给工程执行）

每条需求至少包含以下字段：

- 背景与问题：用户痛点是什么
- 目标与指标：期望改善什么指标
- 优先级：P0/P1/P2
- 方案边界：明确不做什么
- 验收标准：可观测、可测试
- 风险与回滚：失败后的兜底路径

示例（最小模板）：

```md
## 需求：<名称>
- 背景：...
- 目标：...
- 优先级：P1
- 范围内：...
- 范围外：...
- 验收标准：
  - [ ] npm run check 通过
  - [ ] 关键链路烟测通过
  - [ ] 文档同步完成
- 风险：...
- 回滚：...
```

## 8. 运营节奏（建议）

- 每周一次版本评审：核对 roadmap 与实际偏差
- 每周两次需求同步：锁定迭代内范围，禁止临时插单
- 每次合并后更新文档：`technical-documentation.md` 或专项文档
- 每个版本保留一份 QA 结果快照（日期化归档）
