# Runtime3D 与 main 功能对齐说明

## 1. 目的

回答“功能玩法是否变化”这个问题：  
技术栈可替换，但玩法语义不应改变。

## 2. 对齐原则

1. 保持功能语义一致。
2. 保持用户路径一致（可接受 UI 呈现差异）。
3. 技术实现可以完全重构。

## 3. 对齐清单（main -> runtime3d）

| 能力域 | main 基线能力 | runtime3d cleanroom 当前状态 | 差异结论 |
|---|---|---|---|
| 角色系统 | 角色可选与切换 | 已保留；四角色均可切换 | 语义一致 |
| 角色表现 | 2.5D/SVG 表现 | 已接入全角色占位 3D 透视层片 | 表现升级，语义不变 |
| 聊天链路 | 输入-回复闭环 | 已保留主链路 | 语义一致 |
| 四动作 | 聊天/喂食/抚摸/清洁 | 已保留 | 语义一致 |
| 成长照顾 | 状态衰减+成长推进 | 已保留 | 语义一致 |
| 专注模式 | 开始/计时/结束 | 已保留 | 语义一致 |
| 主动行为 | 低状态提醒与主动互动 | 已保留 | 语义一致 |
| 天气系统联动 | 天气与系统状态联动 | 已保留 | 语义一致 |
| 多宠物 | 添加/管理/基础协同 | 已保留 | 语义一致 |
| 托盘热键 | 托盘显隐、热键唤醒 | 已保留 | 语义一致 |

补充说明：
1. 本表用于约束“功能玩法不能缩水”。
2. 当前差异集中在视觉与技术链路，不在玩法语义。

## 4. 当前结论

1. 功能目标：与 main 保持等价，不做缩减。
2. 当前阶段：cleanroom 已完成对齐矩阵与全角色占位 3D 表现，后续继续推进 runtime3d 主线替换与打包收口。
