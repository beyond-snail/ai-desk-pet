# Runtime3D 协作总结与执行规则（2026-03-14）

## 1. 背景总结

在本轮长对话中，出现过“方案讨论”和“工程实现”节奏不同步的问题，导致阶段性偏航。  
核心经验是：先冻结需求，再按任务清单逐项推进，持续更新状态与提交记录。

## 2. 固化后的协作流程

1. 第一步：先产出需求文档（PRD），冻结范围与验收口径。
2. 第二步：再产出开发任务清单（可勾选），按项推进。
3. 第三步：每完成一项即更新清单状态，禁止口头完成不落文档。
4. 第四步：每次非 trivial 改动都执行验证、文档同步、提交与推送。

## 3. 提交与推送规则

1. 每次改动完成后必须 `git commit` + `git push`。
2. 提交信息必须使用中文。
3. 每个提交应绑定任务号或清单项，避免混入无关改动。

## 4. 执行优先级规则

1. 需求文档优先于开发文档。
2. 开发文档优先于临时对话指令。
3. 如临时对话与文档冲突，先更新文档再改代码。

## 5. 对应文档

1. 需求文档：`docs/runtime3d-product-requirements-v1.md`
2. 开发清单：`docs/runtime3d-development-checklist-v1.md`
3. 文档权威入口：`docs/document-authority.md`

## 6. 后续执行约定

1. 新需求先补 PRD，再入开发清单。
2. 任何“做完了”必须落到清单状态和提交记录。
3. 阶段收尾必须产出阶段总结文档，便于下一次交接。
